## v3.x ‚Äî Current

**Packages:** `@randsum/notation`, `@randsum/roller`, all game packages

The notation layer was extracted from `@randsum/roller` into a standalone package, `@randsum/notation`. This package provides:

- `isDiceNotation()` ‚Äî type guard for valid notation strings
- `validateNotation()` ‚Äî structured validation with error details
- `suggestNotationFix()` ‚Äî correction suggestions for invalid input
- `notationToOptions()`, `optionsToNotation()` ‚Äî round-trip conversion between notation strings and options objects
- All shared types: `DiceNotation`, `RollOptions`, `ModifierOptions`, and more

`@randsum/roller` continues to re-export everything from `@randsum/notation` for backward compatibility, so existing import paths do not need to change. Code that imports directly from `@randsum/notation` benefits from a zero-dependency footprint without pulling in the roller's runtime.

Game packages received corresponding minor bumps to stay in sync with the core version.

## v2.x

**Packages:** `@randsum/roller`, initial game packages

The game package ecosystem was introduced in v2. This release added:

- The `createGameRoll` factory in `@randsum/roller` ‚Äî a standardized way for game packages to wrap the core `roll()` function with game-specific validation and result interpretation
- Initial game packages: `@randsum/blades`, `@randsum/daggerheart`, `@randsum/fifth`, `@randsum/pbta`, `@randsum/root-rpg`, `@randsum/salvageunion`
- The `GameRollResult` return type, which wraps roll output with an interpreted outcome tier specific to each game system

Each game package follows the same pattern: accept game-specific input, validate it, translate it to roller options, roll, and interpret the numeric result into a game-meaningful outcome.

## v1.x

**Packages:** `@randsum/roller`

Initial public release of `@randsum/roller`. Established the core API:

- `roll()` ‚Äî the primary entry point accepting numbers, notation strings, and options objects
- Extended dice notation: `NdS` base syntax plus drop (`L`/`H`), keep (`K`/`kl`), cap (`C{...}`), reroll (`R{...}`), explode (`!`), unique (`U`), count successes (`S{...}`), and arithmetic modifiers
- The modifier priority system ‚Äî a fixed execution order ensuring consistent, predictable results regardless of the order modifiers appear in notation
- Zero-dependency design ‚Äî no runtime dependencies
