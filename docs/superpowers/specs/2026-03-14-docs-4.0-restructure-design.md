## Content Strategy

### Per-module getting-started pages

Each follows the same pattern:
1. One-sentence pitch for the package
2. Install command (for notation, clarify when to install standalone vs. just using roller — standalone is for projects that only need parsing/validation without the roll engine)
3. Minimal working example (3-5 lines)
4. "What's next" links to deeper pages in that module

### Per-game pages

Consistent template:
1. Game system summary (what dice mechanics it uses)
2. Install + import
3. Basic roll example with typed result
4. Result interpretation (what outcome tiers mean in that game)
5. Full options/input reference for that game's `roll()`
6. Link back to schema overview ("this game is powered by a `.randsum.json` spec")

### Schema section

All schema and codegen functionality lives within `@randsum/games`. The schema subpath (`@randsum/games/schema`) provides runtime access; the codegen tooling is internal to the games package build.

Narrative arc across four pages:
1. **Overview** — philosophy: "games are data, not code." What `.randsum.json` is, how the internal codegen pipeline turns specs into TypeScript at build time
2. **Reference** — field-by-field schema docs, link to hosted JSON Schema at its `$id` URL
3. **Using loadSpec()** — runtime spec loading via `import { loadSpec } from '@randsum/games/schema'` for dynamic/custom game support
4. **Contributing a Game** — step-by-step: write the spec, validate it against the JSON Schema, run codegen (confirm exact script name from `packages/games/package.json`), test, submit PR

### Welcome section

- **Introduction** — RANDSUM in one paragraph, three bullet points (roller for dice, notation for parsing, games for TTRPG systems), install the one you need
- **Ecosystem Overview** — visual diagram showing dependency flow, brief explanation of each package's role, "where to start" pointers based on use case

### API reference pages

Document all public exports from each package's `index.ts` entry point. Type-focused:
- Exported functions with signatures and brief descriptions
- Exported types/interfaces
- Live REPL examples where CodeExample component already exists

**Roller key exports:** `roll`, `RandsumError`, `RollResult`, `RollOptions`, `ModifierOptions`, modifier types. Note: roller re-exports common notation utilities (`validateNotation`, `isDiceNotation`, `optionsToNotation`) for convenience — document these on the roller API page as re-exports with a link to the notation API reference for full details.
**Notation key exports:** `isDiceNotation`, `validateNotation`, `suggestNotationFix`, `notationToOptions`, `optionsToNotation`, `DiceNotation`, `RollOptions`, `ModifierOptions`, plus additional utilities — see the package's `index.ts` for the full list.

### Tools section

- **Component Library** — consolidates current `tools/components/` pages (index, modifier-reference, roller-playground) into one page
- **Display Utils** — step visualization, modifier docs, and helper functions for generating StackBlitz playground URLs
- **Discord Bot** — usage and invite info
- **Claude Code Skill** — consolidates current `tools/skill/` pages (index, dice-rolling, game-systems, notation) into one page

## Branding

- "Randsum Dice Notation Spec" is the branded name for the notation specification (not "dice notation reference" or similar)

## Sidebar Implementation Note

The sidebar tree above should be translated into Starlight's `items` config format in `astro.config.mjs`. "Schema" and "Game Systems" are label-only sub-groups within the Games section (using Starlight's nested `{ label, items }` pattern). The "Game Systems" group points to flat files under `games/` — there is no `game-systems/` subdirectory.

## Redirects

Explicitly out of scope. The site has very few external users, and this is treated as a fresh launch. No Netlify `_redirects` file or Astro redirect config needed.

## Implementation Order

This should be implemented as an atomic restructure in a single branch. Suggested order within the branch:
1. Sidebar config + empty page scaffolds (site builds and navigates)
2. Welcome section (entry point for all visitors)
3. Roller section (most content can be moved from existing pages)
4. Notation section (move + rebrand existing content)
5. Games section — per-game pages first (updates), then schema sub-pages (new content)
6. Tools section (consolidation of existing pages)
7. Delete old pages

## Out of Scope

- Migration guides (v2→v3, v3→v4) — treating this as a new site
- Changelog page — link to GitHub Releases instead
- Game comparison page — each game page is self-sufficient
- New Astro components or interactive features — reuse existing CodeExample/LiveREPL
- npm README updates (separate effort)
- llms.txt updates (separate effort)
- URL redirects — conscious decision, not an oversight
