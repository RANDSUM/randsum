# RANDSUM 3.0.0 Pre-Release Audit — Round 2

**Date:** 2026-03-09
**Branch:** jarvis/seo
**Auditors:** 6 independent technical analysts (core, games, testing, infra, docs, apps)

---

## Verdict: CLOSER, BUT STILL NOT READY

Round 1 blockers (daggerheart rename, salvageunion size, codecov targets, display-utils build pipeline) are resolved. The ecosystem has improved substantially. However, 8 new blocking issues were found — some introduced by the fixes themselves (CLI workspace dep, codecov overcorrection), some pre-existing and missed (game tsconfig composite, error boundary).

---

## Blocking Issues (Must Fix Before Merging to Main)

### 1. CLI Will Still Fail on npm Publish
**File:** `apps/cli/package.json`

The fix from Round 1 was to make `@randsum/display-utils` public with a build pipeline. That was done. The CLI dependency was never updated:

```json
"@randsum/display-utils": "workspace:~"   // still a workspace ref — breaks on publish
```

Other deps in this file correctly use versioned refs (`"^3.0.0"`). This one doesn't. Consumers of the published CLI will get a broken package.

**Fix:** Change to `"@randsum/display-utils": "^3.0.0"`.

---

### 2. Site: display-utils Docs Page Is Orphaned
**File:** `apps/astro.config.mjs`

`apps/site/src/content/docs/tools/display-utils.mdx` was created and the content is accurate. But there is no sidebar entry for it. Users navigating the site cannot discover or link to it. It exists only if you know the URL.

**Fix:** Add a sidebar entry under "Tools" in `astro.config.mjs`.

---

### 3. Game Packages Missing TypeScript composite
**Files:** `games/*/tsconfig.json` (all 6)

This was flagged in Round 1 and listed as "Should Fix Before Release." It was not fixed. All 6 game packages (blades, daggerheart, fifth, pbta, root-rpg, salvageunion) lack `"composite": true` in their `compilerOptions`. The root roller package has it correctly. Extending a tsconfig does not propagate `composite` — it must be explicit.

**Impact:** TypeScript incremental builds broken across monorepo boundaries. Type errors in game packages won't surface in a `tsc --build` run from root. IDE performance degraded.

**Fix:** Add `"composite": true` to `compilerOptions` in each `games/*/tsconfig.json`.

---

### 4. Component Library Has No React Error Boundary
**File:** `packages/component-library/src/`

`RollerPlayground.tsx` has a try-catch for roller errors in state update logic, but there is no React error boundary anywhere in the component tree. If any React component throws during render, the entire consuming application crashes with no recovery path.

This was flagged in Round 1. It was not fixed.

**Fix:** Wrap `RollerPlayground` and `ModifierReference` in a React error boundary component.

---

### 5. CONTRIBUTING.md References a Directory That Doesn't Exist
**File:** `CONTRIBUTING.md:256`

References `CLAUDE/game-packages.md`, `CLAUDE/scripts.md`, `CLAUDE/modifiers.md`, `CLAUDE/testing.md`. The `CLAUDE/` directory does not exist. Contributors clicking these links get 404s.

This was flagged in Round 1. It was not fixed.

**Fix:** Either create the `CLAUDE/` directory with the referenced guides, or remove the references and point to per-package `CLAUDE.md` files instead.

---

### 6. README Does Not List @randsum/display-utils
**File:** `README.md`

`@randsum/display-utils` is now a public, published package at v3.0.0. It does not appear anywhere in the README. A developer looking at the RANDSUM ecosystem from the repository homepage cannot discover it exists.

**Fix:** Add display-utils to the packages list in README.md.

---

### 7. Codecov 100% Target Will Permanently Fail
**File:** `codecov.yml`

The Round 1 fix raised the coverage target from 99% (which was wrong) to 100% (which overcorrects). `apps/cli/src/tui/hooks/useCursorPosition.ts` reports 13% line coverage and cannot be tested without a full React hook testing environment (`@testing-library/react-hooks`), which bun:test does not support without additional setup. This file will always fail the 100% threshold. Every PR will be blocked by Codecov.

**Fix:** Either (a) add `useCursorPosition.ts` to the Codecov ignore list, or (b) restructure the hook to separate testable logic from React-bound logic, or (c) set the project coverage target to 98% with a narrow threshold.

---

### 8. Publish Script Missing Critical Checks
**File:** `package.json:134`

```json
"publish": "bun run build && bun run test && bun run lint && bun run typecheck && npm publish --workspaces --access=public"
```

Missing: `bun run size` and `bun run check:exports`. CI runs both (lines 66–67 of main.yml), but a developer running the publish script locally bypasses them. An oversized bundle or broken export map could reach npm before CI catches it.

**Fix:** Add `bun run size && bun run check:exports &&` to the publish script.

## Significant Issues (Should Fix Before Tagging 3.0.0)

### display-utils Is Underdressed for a Published Package
**Files:** `packages/display-utils/`

The package was made public and given a build pipeline, but it shipped without:
- `README.md` (roller and notation both have one)
- `CLAUDE.md` (every other package has one)
- npm package metadata: `description`, `keywords`, `engines`, `author`, `license`, `homepage`, `bugs`, `repository` fields in `package.json`
- `files` array only contains `["dist"]` — roller and notation include `src`, `LICENSE`, `README.md`

A consumer discovering this package on npm will see a bare package with no documentation and no description.

---

### display-utils Built for Wrong Target
**File:** `packages/display-utils/bunup.config.ts:10`

```typescript
target: 'node'
```

This library is consumed by `@randsum/component-library`, which is React browser code. Building for Node may apply Node-specific transforms or exclude browser APIs. The correct target for a UI utility consumed in browser contexts is `'browser'`.

---

### DaggerheartAdvantageDisadvantage Type Name Is Still Misleading
**File:** `games/daggerheart/src/types.ts`

The `advantage` field was correctly renamed to `extraDie`. But the exported type `DaggerheartAdvantageDisadvantage = 'Advantage' | 'Disadvantage'` still implies D&D-style advantage (roll 2d20, keep best/worst), when the actual mechanic is adding/subtracting a d6. The package.json description still says "advantage/disadvantage support" without clarification.

Developers from D&D 5e backgrounds will misunderstand the API.

**Fix:** Rename to `DaggerheartExtraDieMode` or similar. Update the package.json description to clarify this is a d6 add/subtract, not a d20-reroll mechanic.

---

### Component Library Has No Component Tests
**File:** `packages/component-library/__tests__/`

Only `tokenize.test.ts` exists — and it tests `@randsum/notation` tokenization, not React components. `RollerPlayground`, `ModifierReference`, and `Overlay` have zero tests. Changes to props, state, event handling, or render behavior are completely unvalidated.

---

### Component Library CSS Injection Requires unsafe-inline CSP
**File:** `packages/component-library/bunup.config.ts`

`css: { inject: true }` injects styles via runtime DOM manipulation, requiring `style-src 'unsafe-inline'` in any Content Security Policy. This makes the component library unusable in serious enterprise deployments and Next.js/Remix apps with strict CSP. No escape hatch to opt out.

This was flagged in Round 1 as a "Significant Issue." It remains unfixed.

---

### Property-Based Tests Are Not Property Tests
**Files:** `games/*/__tests__/*.property.test.ts`

These files use the `.property.test.ts` naming convention and `fast-check`, but test no properties. They verify that functions return values from a known set, or that totals fall within an expected range — basic smoke validation. No commutativity, invariant, or distribution tests. They provide false confidence without meaningful coverage.

---

### No Cross-Package Integration Tests
No test file in the monorepo exercises two packages together. A breaking change in `@randsum/roller` that passes all roller tests could silently break all game packages. Game packages mock their roller dependency in tests, so they wouldn't catch it either.
