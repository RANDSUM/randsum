# RANDSUM 3.0.0 Pre-Release Audit

**Date:** 2026-03-09
**Branch:** jarvis/seo
**Auditors:** 6 independent technical analysts (core, games, testing, infra, docs, apps)

---

## Verdict: NOT READY

The core engine is excellent. The surrounding ecosystem has blocking issues that will cause failures at publish time or embarrass us in production. These are solvable — most in an afternoon — but they need to be addressed before shipping 3.0.0.

---

## Blocking Issues (Must Fix)

### 1. CLI Will Fail on npm Publish
**File:** `apps/cli/package.json` lines 44–48

```json
"@randsum/display-utils": "workspace:~",  // PRIVATE, unpublished
"@randsum/notation": "^3.0.0",            // npm version
"@randsum/roller": "^3.0.0",              // npm version
```

`@randsum/display-utils` is marked `private: true` and has no build pipeline (`main` points to `src/index.ts`, no `dist/`). The workspace protocol resolves only within the monorepo. After publish, consumers will get a broken package.

**Fix:** Either inline display-utils into the CLI, or give display-utils a proper build pipeline and publish it. Do not publish the CLI in this state.

---

### 2. Codecov Config Contradicts Documented Policy
**File:** `codecov.yml` lines 4–10

```yaml
target: 99%
threshold: 2%
```

CLAUDE.md states: "Coverage target: 80% project, 70% patch." The codecov config demands 99%, which will either permanently fail CI or has already been silently ignored. One of these is the source of truth — pick one and reconcile.

**Fix:** Update `codecov.yml` to match CLAUDE.md (80% project, 70% patch).

---

### 3. Discord Bot Has Zero Tests
**File:** `apps/discord-bot/` — no `__tests__/` directory

This is a production service making external API calls. Every slash command handler has untested error paths. The notation validation in `commands/roll.ts:23–37` is a try/catch with no test coverage. No rate limit handling. Graceful shutdown is missing. If it breaks in production there is no regression suite to catch regressions.

This is acceptable for a weekend project. It is not acceptable in a v3.0.0 release ecosystem.

**Fix:** Add at minimum happy-path tests for each command. Add rate limit handling before any significant traffic.

---

### 4. Salvageunion Bundle at 98.4% of Size Limit
**File:** `package.json` line 63–64

```json
{ "path": "games/salvageunion/dist/index.js", "limit": "240 KB" }
```

Current size: ~236 KB. Buffer: ~4 KB. The next bug fix that touches this package may break CI. Additionally, CLAUDE.md documents the limit as "170KB+" but the actual limit in `package.json` is 240 KB — the documentation is wrong.

**Fix:** Raise the limit to 250 KB or 280 KB to give real headroom. Update CLAUDE.md to reflect the correct number.

---

### 5. Daggerheart Type Naming Is Misleading
**File:** `games/daggerheart/src/types.ts` line 23

The return type uses a field called `advantage` that holds the result of **both** advantage and disadvantage rolls. When disadvantage is used, this field contains a negative value. Nothing in the type signature communicates this. Consumers calling `.advantage.roll` expecting a positive number will get burned.

**Fix:** Rename the field to `advantageDisadvantage` or `sideEffect`, or use a discriminated union that makes the sign explicit.

---

### 6. Salvageunion Breaks the Ecosystem Pattern
**File:** `games/salvageunion/src/roll/index.ts` (lines 11–33 per game package audit)

Every other game package uses `createGameRoll`. Salvageunion calls `coreRoll()` directly and returns a type with inconsistent generics (`RollRecord<SalvageUnionTableListing | string>` vs other packages using plain `RollRecord`). It also exports `SALVAGE_UNION_TABLE_NAMES`, a configuration constant — no other game package exports configuration.

This package is technically functional but architecturally inconsistent with the ecosystem it claims to belong to. At minimum, document why it's different. At best, refactor it to use `createGameRoll`.

Additionally, its tests mock the external `salvageunion-reference` library entirely, meaning the integration between this package and its data source is not actually tested. If `salvageunion-reference` changes its table structure, this package breaks silently.

## Significant Issues (Should Fix Before Release)

### Component Library CSS Injection Is a CSP Problem
**File:** `packages/component-library/bunup.config.ts` (`css: { inject: true }`)

CSS injected via `innerHTML` into `<style>` tags requires `unsafe-inline` in Content Security Policy headers. This makes the component library unusable in any environment with a reasonable CSP — most enterprise apps, any serious Next.js/Remix deployment. There is no escape hatch to opt out of injection.

There is also no React error boundary around the roller logic. If `@randsum/roller` throws an uncaught exception (which it does on invalid input), the entire consuming app crashes.

**Fix:** Emit CSS as a separate file consumers can import. Wrap components in an error boundary.

---

### TypeScript Composite Missing from All Game Packages
**Files:** `games/*/tsconfig.json`

All game packages reference `packages/roller` but none have `"composite": true` in their tsconfig. This means TypeScript project-level incremental builds are broken for games — type errors in games won't surface in a `tsc --build` run from root. The root `tsconfig.json` also doesn't include games in its `references` array.

**Fix:** Add `"composite": true` to all game package tsconfigs and add them to the root `tsconfig.json` references.

---

### Publish Script Missing Pre-Flight Checks
**File:** `package.json` line 130

```json
"publish": "bun run build && bun run test && bun run lint && bun run typecheck && npm publish --workspaces --access=public"
```

Missing: `bun run size` (could publish an oversized bundle), `bun run check:exports` (could publish with broken exports), no git tag creation, no changelog verification. This is the single command that puts packages on npm — it should be the most defensive script in the repo.

---

### Site May Have Broken Routing
**File:** `apps/site/src/pages/index.astro`

The site uses Starlight (Astro docs theme). Depending on i18n configuration, routes may need to be prefixed with `/en/` (e.g., `/en/packages/roller/` vs `/packages/roller/`). The homepage hardcodes paths like `/tools/skill/` for which no corresponding page exists. This should be verified with `bun run site:build` and link-checked before release.

---

### CONTRIBUTING.md References Nonexistent Directory
**File:** `CONTRIBUTING.md` line 256

References `CLAUDE/game-packages.md`, `CLAUDE/scripts.md`, etc. The `CLAUDE/` directory does not exist. Either create the guides or remove the references.

## What Is Genuinely Good

The core engine is well-built. Specific praise:

- **roller + notation:** 914 passing tests. Zero `any` casts. No unsafe coercions. Modifier priority system is explicit and tested. Explosion depth limits prevent infinite loops. ReDoS-resistant parser with length limits. All documented APIs are actually exported.
- **Game package pattern (5 of 6):** Consistent use of `createGameRoll`, consistent input validation with accurate game-specific ranges, consistent return type structure.
- **Error hierarchy:** `RandsumError → ModifierError | ValidationError | RollError | NotationParseError` is clean and testable.
- **Lefthook hooks:** Pre-commit auto-fix + pre-push full test suite is exactly right.
- **CI matrix:** Ubuntu + macOS, with security audit on schedule.
- **Dice notation spec:** One of the best DSL documentation files in any open source dice library. Accurate, comprehensive, well-exemplified.
- **Bundle discipline:** 8 of 9 packages within healthy margins. Salvageunion's size is justified by the embedded table data.

## Test Suite Summary

- **roller:** 553 tests — excellent
- **notation:** 361 tests — excellent
- **Game packages:** 59 tests each (mechanically correct; game rule semantics underspecified but acceptable)
- **CLI:** ~6 test files — adequate for utilities, missing error paths
- **Component library:** 1 test file — insufficient
- **Discord bot:** 0 — not acceptable for production
- **Integration tests:** None — no end-to-end coverage across packages

The property-based tests are meaningful, not trivial. The 9999-iteration stress tests are appropriate. Seeded RNG usage is correct throughout.
